## 1. 概要

冷蔵庫にある食材を起点に献立を AI が提案し、不足食材だけを買い物リスト（iOS=リマインダー / Android=Google ToDo）へ自動追加するアプリ。在庫を「使い切る」方向に働き、買いすぎ・使い忘れを減らす。Healsio / Hotcook 連携で調理モード・時間まで提案するのが差別化点。

- アプリ名: つかいきり / Tsukaikiri
- アプリ ID: `com.futo4.tsukaikiri`
- 対象: macOS + iOS 26 + Android（最低 API は Flutter 既定）。**スマホ・タブレット両対応**（画面幅で分岐）
- 開発順: macOS で縦通し → iOS → Android

### 基本フロー
1. 冷蔵庫の在庫を見る（賞味期限が近いものを優先）
2. その在庫で作れる献立を AI が提案
3. 在庫が乏しければ AI が補完献立を提案
4. 必要で家にない材料を買い物リストへ自動追加
5. 調理したら使った食材を在庫から減算

## 3. 在庫管理（✅ 最小版 実装済み）

### 3.1 食材の項目
| 項目 | 内容 |
|---|---|
| 名前 | 表示名（ユーザー入力、翻訳しない） |
| カテゴリ | 下記12種から選択 |
| 数量 | **小数許可**（例 0.5 / 1.5）。0 以下は登録不可 |
| 単位 | 固定セットから選択、または**カスタム自由入力** |
| 賞味期限 | **任意**。カテゴリ目安で初期値を自動セット、手動修正可、空も可 |

### 3.2 カテゴリ（12種・この表示順）
肉 / 魚 / 野菜 / 果物 / 乳製品 / 卵 / 穀物・主食 / 調味料 / 冷凍食品 / 飲料 / 常備品 / その他

- ラベルは i18n 対象（ja/en）。内部キーは言語非依存（列挙子名で保存）

### 3.3 単位
- 固定セット: 個 / g / kg / ml / L / 本 / 枚 / パック / 袋 / 合 / 杯 / 缶
- いずれにも当てはまらない場合は「カスタム…」を選び自由入力
- 固定単位はラベルをローカライズ。カスタムは入力文字をそのまま表示

### 3.4 賞味期限の扱い
- **カテゴリ別の日持ち目安テーブル**（最小手書き版）を同梱し、`登録日 + 目安日数` を初期値にセット
  - 目安日数: 肉3 / 魚2 / 野菜5 / 果物7 / 乳製品7 / 卵14 / 穀物・主食30 / 調味料180 / 冷凍90 / 飲料14 / 常備品180 / その他=目安なし
  - 「その他」は目安なし＝期限を自動セットしない
- ユーザーが日付を手で触ったら、その後のカテゴリ変更では上書きしない
- すべて「目安」であり厳密な規則ではない（手動修正前提）
- 食材名単位の個別ルール・保存状態（冷蔵/冷凍/常温）・USDA FoodKeeper 取り込み・AI補完は 🔜 後フェーズ

### 3.5 期限の警告表示
| レベル | 条件 | 色 |
|---|---|---|
| 期限切れ | 残り < 0 日 | 赤（error） |
| 当日 | 残り = 0 日 | 赤（error） |
| 警告 | 残り 1〜3 日 | オレンジ |
| 通常 | 残り 4 日以上 | 控えめ |
| 期限なし | 日付未設定 | グレー |

- **期限切れ品は一覧から自動削除せず、赤表示のまま残す**

### 3.6 一覧・操作
- 並び順: 期限が近い順 → 期限なしは末尾 → 同順なら更新日時の新しい順
- カテゴリフィルタ（「すべて」＋12種）
- 減算: ワンタップで −1。**0 になっても在庫から自動削除しない**（明示削除のみ）
- 追加・編集フォーム、削除（確認ダイアログあり）

### 3.7 名寄せ（normalizedName）
- 言語非依存の名寄せキー。AI 連携前に登録した食材は **name をそのまま流用**している
- **バックフィル（実装済み）**: 未付与の行だけを対象に AI でキーを一括付与する冪等なサービスを用意。実行タイミングは献立提案フローの後を想定（UI 接続時に配線）
- 期限の引き当ては当面カテゴリ目安のみ

## 4. 多言語対応 (i18n)（✅ 実装済み）

- 対応言語: **日本語 / 英語**
- UI 文言は `flutter_localizations` + ARB（`lib/l10n/app_ja.arb` / `app_en.arb`）
- 言語決定: 既定は端末ロケール追従。設定画面で「システムに従う / 日本語 / English」を選択（`localePref` に保存）
- AI 出力言語もこの設定に連動（🚧 `RecipeConstraints.outputLocale` がプロンプトに反映される実装済み。UI からの接続は献立提案フェーズで）
- カテゴリ名・単位ラベルはローカライズ。ユーザー入力の食材名は翻訳しない

---

## 5. オフライン時の挙動（方針）

- 在庫管理（ローカルDB）はオフラインで通常動作
- 通信必須機能（AI献立・カメラ登録・買い物リスト・同期）は、**実行 → 失敗時に「電波の良い場所か Wi-Fi へ」表示**。事前判定でボタンを無効化しない

## 6. 後続機能の仕様概要

### 6.1 買い物リスト連携（🚧 サービス層実装済み）
- 献立に必要 & 在庫にない材料を抽出し、選択中リストへ追加
- **不足判定（確定）**: 在庫扱いは数量 > 0 のみ。名前一致（大文字小文字無視）→ 名寄せキー一致の順で照合し、表記ゆれ（鶏むね/鶏胸肉）は AI 名寄せで吸収。複数献立の同一材料は1件に集約
- **数量の載せ方（確定）**: リマインダーのタイトルは材料名のみ、分量と由来献立名は notes（タイトル完全一致の重複チェックを活かすため）
- **重複チェック（確定）**: 追加先リストの**未完了**項目とタイトルを小文字化して完全一致比較し、重複はスキップ。追加結果は実追加件数で通知
- macOS/iOS=リマインダー（EventKit、自前 platform channel）/ Android=Google ToDo（Google Tasks API）🔜
- リスト一覧取得・新規リスト作成（iCloud 優先）・項目追加・アクセス権限リクエストまで実装済み
- 保存するのは表示名でなく**識別子**（macOS/iOS=calendarIdentifier / Android=tasklist id）
- 買ったものをまとめて在庫へ戻すショートカット（❓実装方式未定）

### 6.2 AI 献立提案（🚧 全4社実装済み）
- 在庫を渡して作れる献立を提案（既定3案）。期限間近（残り2日以内）の食材を優先するようプロンプトで強調
- 在庫が乏しければ補完献立＋不足材料を提示
- 「主菜だけ」「あと1品」等の条件指定（`extraRequest` としてプロンプトに反映）
- 所有家電に応じた手順（Hotcook モード・時間 / Healsio 温度 / 未所有は通常調理）
- レスポンスは **JSON 固定**（前置き・コードフェンス禁止＋API 側でも JSON 出力を強制）。フィールド名は言語固定、値のみ指定言語。スキーマは設計書 §5.1 参照
- 対応プロバイダ（優先順位順）: Gemini / Grok / OpenAI / Claude。API キーはユーザーが各社から取得して登録
- **使用モデルはユーザーが選択**: 各社のモデル一覧 API から取得した候補を設定画面で選び、プロバイダごとに保存（ハードコードしない）。未選択時のフォールバック: gemini-2.0-flash / grok-4.3 / gpt-5-mini / claude-opus-4-8（2026-06 時点）
- 通信タイムアウト 60 秒・自動リトライなし（失敗時はユーザーが再試行）

### 6.3 カメラ登録（半自動）
- カメラ/写真ライブラリから1〜最大10枚 → Vision で食材候補（名前・推定個数・単位・確信度 0.0〜1.0）抽出（🚧 認識 API は実装済み）
- 低確信度の候補は確認画面で薄く表示
- **必ず確認画面を経由**してから在庫登録（AI出力を直接書かない）🔜
- 送信前に画像を縮小（コストガード、❓具体値未定）

### 6.4 データ同期
- iOS=iCloud / Android=Google Drive App Data（ユーザー自身のクラウド・無料）
- 当面は丸ごとバックアップ/復元

### 6.5 ヘルプ / このアプリについて
- 賞味期限データの出典（USDA/FSIS FoodKeeper）を明記、和食材は補完、表示は目安である旨
- 「期限はいつでも手動修正可」を案内

### 6.6 ドネーション
- メニュー「作者をサポート」→ `url_launcher` で Buy Me a Coffee を外部リンクで開くだけ（IAP不使用、全機能無料・機能差なし）
- ❓ URL は後で作成・差し込み
