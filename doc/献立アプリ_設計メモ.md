# つかいきり（Tsukaikiri）設計メモ

冷蔵庫起点・献立 & 買い物リストアプリ

| 項目 | 値 |
|---|---|
| アプリ名 | つかいきり |
| 英語表記 | Tsukaikiri |
| Flutter プロジェクト名 / package | `tsukaikiri`（lower_case_with_underscores） |
| Git フォルダ / リポジトリ名 | `tsukaikiri`（小文字） |

> 名前の由来: 在庫を「使い切る」方向に働くアプリの性格をそのまま表したもの。App Store / Google Play / Mac App Store で同名アプリは確認されず（2026年6月時点）。ただし冷蔵庫食材→レシピ提案ジャンル自体は競合が多いため、差別化は Hotcook / Healsio 連携と、買い物リスト自動連携（macOS/iOS=リマインダー / Android=Google ToDo）で行う。

## コンセプト

冷蔵庫にある食材を起点に献立を組み立て、足りないものだけを買い物リスト（macOS/iOS=リマインダー / Android=Google ToDo）に自動で追加するアプリ。在庫を使い切る方向に働くので、買いすぎ・使い忘れが減る。Healsio / Hotcook での調理を前提に、調理モードや時間まで提案に含める（所有家電に応じて出し分け）。

**AI は既定でオンデバイス（端末内）で動く。** インストールしてすぐ・API キー不要・無料・オフライン・完全プライベートで使えることを既定とする（対応端末の場合）。より賢い提案がほしい上級者は、任意で自分の API キー（Claude / OpenAI / Gemini / Grok）を登録して上位のクラウドモデルに切り替えられる。詳細は「AI プロバイダ」を参照。

基本の流れ:

1. 冷蔵庫の在庫を見る（特に賞味期限が近いものを優先）
2. その在庫で作れる献立を AI が提案
3. 在庫が乏しい / 作れるものがなければ、AI が新しい献立を提案
4. 献立に必要で家にない材料を「欲しいもの」リストへ自動追加
5. 調理したら、使った食材を在庫から減算

## 主要機能

### 1. 冷蔵庫在庫管理

- 食材の登録: 名前、数量（個数を基本に、単位を選択可。既定「個」、調味料・米などは g/ml/本/合 などに変更）、カテゴリ（肉・魚・野菜・乳製品・調味料・常備品 など）、賞味期限（任意・下記の方式で初期値を自動設定）
- 賞味期限が近いものをトップに表示 / バッジ表示
- 献立で使った食材をワンタップで減算、買ったものを在庫に追加
- 名寄せ: 「鶏むね」「鶏胸肉」などの表記ゆれを AI 側で吸収して同一視
- **カメラ登録（半自動）**: 冷蔵庫内を撮影 → AI が食材候補を出す → ユーザーが確認・修正して登録（下記）

#### 賞味期限の決め方

- **カテゴリ別の日持ち目安テーブル**を持ち、「登録日 + 目安日数」で初期値を自動セット（例: 鶏むね肉=冷蔵2日、卵=2週間 など）
- カメラでのパッケージ日付読み取りは精度が不安定なため**採用しない**
- ユーザーがいつでも手入力で修正できる
- 賞味期限は**任意**。調味料・常備品など管理不要なものは空のままでよい

**目安テーブルの作り方（同梱データ）**

- **ベースに USDA FoodKeeper Data を採用**。Data.gov で公開（`catalog.data.gov/dataset/fsis-foodkeeper-data`）、650 品目超を冷蔵/冷凍/常温の保存日数つきで収録。米政府の公開データで利用ハードルが低い。`ShelfLifeRule`（key / days / storage）にほぼそのままマッピングできる
- ただし**米国基準・英語・米国分類**で、和食材（油揚げ・絹ごし豆腐・ちくわ等）は手薄。英語品目→日本語へのマッピングが必要
- **和食材・日本特有の品目は AI で目安を生成して補完**し、同梱データにマージ
- 取り込んだ値はすべて「目安」であり厳密な規則ではない（FoodKeeper 自身も目安と明記）。自動セット＋手動修正の前提と噛み合う
- 実装手順の目安: ①FoodKeeper データを取得・整形 → ②日本語マッピング → ③不足する和食材を AI で補完生成 → ④アプリ同梱の `ShelfLifeRule` データとして固める（毎回 API を呼ばず、ビルド時に1度作る）
- **データ出典はアプリ内ヘルプに明記する**（「ヘルプ / このアプリについて」機能を参照）

#### カメラ登録のフロー（半自動）

1. カメラ or 写真ライブラリから1〜複数枚撮る（`image_picker` / `camera`）
2. 画像を Vision 対応モデルに渡し、写っている食材を JSON 配列で返させる（各要素: 名前・推定個数・確信度）
3. 返ってきた候補を**確認画面**に並べる（チェックで採用/除外、個数・単位・カテゴリを修正可）
4. 確定したものだけ在庫に登録。名寄せも通す
- **原則 AI 出力を直接在庫に書かず、必ず人が一度確認する**（誤認識を「直す」で吸収。庫内は重なり・容器・複数個で誤りやすい前提）
- 確信度の低い候補は薄く表示するなど、確認の負担を下げる UI にする
- **コストガード**: 送信前に画像を縮小（長辺を一定サイズに圧縮）してから渡す。1回の認識で**最大10枚**まで。Vision 課金と通信量を抑える

### 2. 献立提案（在庫逆算 → なければ AI 提案）

- 在庫リストを AI に渡し「これで作れる献立」を提案させる
- 賞味期限が近い食材を優先的に使うよう指示
- 在庫だけでは厳しい場合、AI が補完献立を提案し、不足材料を洗い出す
- 各献立に、所有家電に応じた手順を添える（Hotcook / Healsio のモード・加熱時間、未所有なら通常調理）
- 「あと1品ほしい」「主菜だけ」など条件指定にも対応

### 3. 買い物リスト連携（OS 別: リマインダー / Google ToDo）

- 献立に必要 & 在庫にない材料を抽出
- 選択中のリストへ自動追加（重複チェックあり）
- 追加した材料を、買い物完了時にまとめて在庫へ戻すショートカット

### 4. 作者をサポート（ドネーション）

- メニューに「作者をサポート」項目を置き、タップで **Buy Me a Coffee** の外部ページを開く（≒1ドルの応援）
- **外部リンク方式**（アプリ内課金=IAP は使わない）。タップで外部ブラウザ / アプリ内ブラウザ（`url_launcher`）で開くだけ
- 決済・個人情報の処理はすべて Buy Me a Coffee 側で完結。アプリ自身は金銭・個人データを一切扱わない

### 5. ヘルプ / このアプリについて

- 賞味期限の**データ出典を明記**: 保存期間の目安は米農務省（USDA/FSIS）の公開データ **FoodKeeper Data** をベースにしていること、和食材は補完していること、表示はあくまで目安で実際とは異なりうること、を記載
- FoodKeeper のリンク（Data.gov）を載せ、出典をたどれるようにする
- あわせて「賞味期限はいつでも手動修正できる」旨をヘルプに案内
- **AI について案内**: AI は端末内（オンデバイス）で動作し、対応端末なら無料・オフラインで使えること、写真や在庫データは端末外に送られないこと（プライバシー）、より賢い提案がほしい場合は設定の「詳細」で自分の API キーを登録できることを記載
- i18n 対象（日本語 / 英語 / スペイン語）。出典名・リンクは共通、説明文はローカライズ

## 技術選定の根拠（なぜ Flutter か）

2026 年時点で、個人開発・iOS/Android 両対応・1人で保守、という条件下では Flutter が最もコスパが高いと判断。

- **両 OS を1コードベース＋UI まで共通化**できる。Kotlin Multiplatform はロジックは共有できるが UI をネイティブで2回書く必要があり、個人には重い。
- **このアプリのネイティブ依存は買い物リスト連携だけ**。コアは AI（HTTP 呼び出し）とローカル DB で、いずれも OS 非依存。ネイティブ部分は platform channel で薄く書く設計に既になっており、Flutter の弱点（ネイティブ機能アクセス）が最小限に収まる。
- **Claude Code との相性**: Dart/Flutter は学習データが豊富で、生成 AI が安定して書ける。
- **将来性**: Flutter は 2026 年も活発（GitHub 約 17.5 万スター、BMW・Alibaba・Toyota 等が本番採用）。Google 自身が NotebookLM など新規 AI 系アプリで Flutter を採用しており、当面の投資継続が見込める。

留保点（今回は当てはまらない）: OS 最新機能を即日フル活用したい、重いネイティブ SDK を多用する、といったケースではネイティブや KMP が有利になる。本アプリはどちらも該当しない。

代替案の位置づけ: React Native（JS/TS 資産がある人向け）、KMP（UI ネイティブ・シェア急伸中）、ネイティブ2本（最高体験だが個人保守は重い）。シェアは各社拮抗しており、Flutter が廃れた状況ではない。

### データ同期（プラットフォーム分岐）

在庫データはローカル DB が主。機種変更で消えないよう、ユーザー自身のクラウドに同期する。共通インターフェース `SyncService` を1つ切り、OS で実装を差し替える。

| OS | 同期先 | 実装方式 |
|---|---|---|
| macOS | iCloud | iCloud Documents（または CloudKit）。ユーザーの iCloud 容量を使用。iOS と実装共通 |
| iOS | iCloud | iCloud Documents / key-value（または CloudKit）。ユーザーの iCloud 容量を使用 |
| Android | Google Drive **App Data フォルダ** | googleapis（`drive.appdata` スコープ）。ユーザーから見えない隠しフォルダ。WhatsApp の Android バックアップと同方式。既製パッケージ（`local_first_gdrive_backup` 等）あり |

- どちらも**保存先はユーザーのクラウド容量なので開発者コストはゼロ**
- Android は Google Cloud Console で Drive API 有効化 + OAuth クライアント（SHA-1 付き）が必要。買い物リストの Google Tasks と同じ Google 連携なのでまとめて準備できる
- 同期方式は当面「丸ごとバックアップ/復元」で十分（在庫データは小さい）。差分同期は将来拡張

### オフライン時の挙動

- **在庫管理（ローカル DB）はオフラインで通常通り動く**
- 通信必須の機能（献立提案・カメラ登録・買い物リスト連携・データ同期）は、通信を試みて失敗したら**「電波状態の良い場所で使うか Wi-Fi に接続してください」とエラーメッセージを表示**
- 事前のオフライン判定でボタンを無効化するより、実行 → 失敗時にメッセージ、のシンプルな方式でよい

### レイアウト方針（スマホ / タブレット / デスクトップ）

macOS・iPad・Android タブレットにも対応する。OS 別ではなく**画面幅のブレークポイントで分岐**する（同一コードベース）。

- `LayoutBuilder` / `MediaQuery` で narrow / wide を判定。OS（macOS/iOS/Android）で作り分けはせず、幅で出し分ける
- **スマホ・狭い幅**: 単一ペインの画面遷移（一覧 → タップで詳細へ）
- **macOS・タブレット・広い幅**: 常時二ペイン（master-detail）。例: 左に在庫一覧／右に詳細・編集、左に献立候補／右にレシピ詳細
- macOS はウィンドウのリサイズに追従（幅が狭くなれば単一ペインに切り替え）
- 縦横回転に追従。タブレットで縦持ちのときは単一ペインに切り替えるなど、幅基準で素直に
- Flutter は画面サイズに応じて同じウィジェットを再配置できるため、レイアウトの出し分けは追加コストが小さい

### 買い物リスト連携（プラットフォーム分岐）

「リスト一覧を取得 → ユーザーが選ぶ or 新規作成 → 以降そこへ書き込む」という共通の体験を、OS ごとに別実装で実現する。共通インターフェース（`ShoppingListService`）を1つ切り、iOS / Android で実装を差し替える。

| OS | 連携先 | 実装方式 |
|---|---|---|
| macOS | Apple リマインダー | EventKit（macOS でも利用可）。platform channel で Swift/ObjC 側を呼ぶ。iOS と実装をほぼ共有できる |
| iOS | Apple リマインダー | EventKit。Flutter からは platform channel で Swift 側を呼ぶ（or 既存プラグイン） |
| Android | Google ToDo（Google Tasks） | Google Tasks API（OAuth 2.0、スコープ `auth/tasks`） |

- 共通 protocol 例: `getLists()` / `createList(name)` / `addItems(listId, items)`
- macOS / iOS の EventKit は Dart から直接触れないため、platform channel で薄い Swift ブリッジを書くか、既存の reminders プラグインを使う
- Android の Google Tasks は OAuth が必要。`google_sign_in` でトークンを取り、Tasks REST API を叩く
- **配布時の注意**: Google Tasks のような個人データスコープを公開アプリで使うと OAuth 検証プロセスが必要。個人利用（自分のアカウントのみ）なら未検証のまま使える

### 多言語対応 (i18n)

日本語 / 英語に対応。最初から組み込む（後付けは文字列の洗い出しが大変なため）。

**(1) UI 文言**
- `flutter_localizations` + ARB ファイル（`app_ja.arb` / `app_en.arb`）で管理。Flutter 公式の仕組み
- 言語の決定: 既定は端末のロケールに追従。設定画面で「日本語 / English / システムに従う」を手動選択でき、`UserSettings.locale` に保存
- カテゴリ名などの固定ラベルはローカライズ対象。ユーザーが入力した食材名はそのまま保持（翻訳しない）

**(2) AI 献立提案の出力言語**
- 表示言語に合わせて AI にもその言語で返させる。`RecipeConstraints` に `outputLocale` を含め、プロンプトで「○○語で返答」と明示
- 名寄せの `normalizedName` は言語非依存のキーとして扱う（表記ゆれ吸収が目的なので、UI 表示言語が変わってもキーは安定させる）
- JSON スキーマ（フィールド名）は言語によらず固定。値（title・steps など自然文）だけを指定言語で生成させる

### ドネーション（外部リンク方式）の実装メモ

- メニュー項目「作者をサポート」→ `url_launcher` で Buy Me a Coffee の URL を開くだけ。アプリ側は決済を一切持たない
- **なぜ IAP を使わないか**: Apple は「承認された非営利団体でない限り、アプリ内（IAP）で寄付・募金を集めることを禁止」しており、募金系アプリは無料配布のうえアプリ外（Safari 等）でのみ集金可。個人開発者の寄付は IAP に乗せられない
- 一方、米国ストアフロントでは外部決済への誘導ボタン・リンクの制約が撤廃されており、**外部リンクでの「作者をサポート」は問題なく置ける**
- **審査での書き方**: 「digital content/feature のアンロックではない」ことを明確に。寄付しても機能差が出ない（全機能が無料で使える）設計にしておくと審査が通りやすい
- プライバシー表示: アプリはデータ収集なし。遷移先（Buy Me a Coffee）は外部サービスである旨を一言添えると親切

## データモデル（たたき台）

```
Ingredient（食材）
  - id
  - name              // 表示名
  - normalizedName    // 名寄せ用キー（AIが付与）
  - category          // 肉/野菜/調味料 など
  - quantity          // 数値（個数を基本に持つ）
  - unit              // 単位（既定「個」。g / ml / 本 / 合 などユーザー設定可）
  - expiryDate        // 賞味期限（任意・カテゴリ目安から初期値を自動セット、手動修正可）
  - updatedAt

Recipe（献立・提案結果）
  - id
  - title
  - ingredients[]     // {name, amount} のリスト
  - appliance         // Hotcook / Healsio / なし
  - cookMode          // 調理モード
  - cookMinutes
  - steps[]
  - usesExpiringSoon  // 期限間近の食材を使うか
  - generatedBy       // 提案に使ったプロバイダ/モデル名（記録用）

UserSettings（アプリ設定・単一レコード）
  - shoppingListId    // 書き込み先リストの識別子（iOS=calendarIdentifier / Android=Google Tasks の tasklist id）
  - shoppingListName  // 表示用（UIに出す現在のリスト名）
  - selectedProvider  // 選択中の AI プロバイダ。「オンデバイス既定」または段階2の自前キー（claude/openai/gemini/grok）のいずれかを表す。未設定はオンデバイス既定に解決
  - locale            // UI 言語（ja / en / system）。AI 出力言語にも反映
  - syncEnabled       // クラウド同期の ON/OFF（iOS=iCloud / Android=Google Drive）
  - lastSyncedAt      // 最終同期日時（任意）
  - appliances[]      // 所有する調理家電（下記 Appliance）

Appliance（所有家電）
  - type              // enum: hotcook / healsio
  - series            // ホットクック: pro / with ・ ヘルシオ: AX系 など（任意）
  - capacity          // 容量（2.4L / 1.6L / 1.0L など、任意）
  - modelNumber       // 型番（任意・自由入力可）

ShelfLifeRule（日持ち目安・アプリ同梱データ）
  - key               // カテゴリ or 正規化食材名（normalizedName と対応）
  - days              // 登録日からの目安日数
  - storage           // 冷蔵 / 冷凍 / 常温（任意）
  // 登録時に該当ルールを引いて expiryDate の初期値を計算。食材名優先、なければカテゴリでフォールバック
```

## 主要な画面 / フロー

0. **初回オンボーディング**（初回起動時のみ）
   - **AI は初回フローで API キーを入力させない**（キーなしで即使える前提）。オンデバイス AI が使えるかだけを判定し、使えない場合のみ案内する（Android 非対応端末なら「この端末では献立提案・カメラ登録は使えない」旨と、在庫管理などは使える旨を表示）。自前キーの登録は設定の「詳細」へ誘導するに留める
   - 買い物リスト連携の許可・接続（macOS/iOS=リマインダーのアクセス許可 / Android=Google サインイン）
   - 連携先のリスト一覧を取得して提示（macOS/iOS=リマインダーのリスト / Android=Google Tasks のリスト）
   - ユーザーが「既存リストから選ぶ」or「新規リストを作成」を選択
   - 選んだリストの識別子を保存（以降の書き込み先）
   - **調理家電の選択**: 「ホットクックを持っているか」「ヘルシオ（ウォーターオーブン）を持っているか」を選ぶ。持っている場合は機種（シリーズ・容量）も選択
   - すべて設定画面からあとで変更可
1. **在庫画面**: 食材一覧（期限近い順）、追加・編集、カテゴリフィルタ。カメラ登録への導線
2. **カメラ登録画面**: 撮影 → AI が候補を抽出 → 確認画面でチェック/修正 → 確定して在庫へ
3. **献立提案画面**: 「在庫から提案」ボタン → 候補が数件出る → 1つ選ぶ
4. **買い物リスト確認**: 不足材料の一覧 → 「選択中のリストに追加」ボタン
5. **調理完了**: 使った食材にチェック → 在庫から減算

### 買い物リスト連携の実装メモ

共通インターフェース `ShoppingListService`（`getLists` / `createList` / `addItems`）を1つ切り、OS で実装を差し替える。

- **macOS / iOS（リマインダー）**: EventKit（macOS / iOS 共通）。取得は `calendars(for: .reminder)`、新規作成は `EKCalendar(for: .reminder)` を iCloud ソースに紐付けて保存。Dart から直接触れないので platform channel で薄い Swift ブリッジを書くか、既存プラグインを使う。macOS と iOS で同じ Swift コードを共有できる
- **Android（Google ToDo）**: Google Tasks API。`google_sign_in` で `auth/tasks` スコープのトークンを取得し、`tasklists.list` / `tasks.insert` を REST で呼ぶ
- **共通**: 保存するのは表示名ではなく**リストの識別子**（macOS/iOS=`calendarIdentifier` / Android=tasklist id）。表示名はリネームで変わるため。識別子で引き当て、UI には現在の表示名を出す
- 保存先が見つからない場合（リスト削除など）は、再度リスト選択に誘導

## AI プロバイダ（2段構え: オンデバイス既定 + 任意の自前キー）

献立提案・材料の名寄せ・調理手順生成・カメラ食材認識に LLM を使う。一般ユーザーが API キーを取得・入力するのは離脱要因になるため、**オンデバイス AI を既定**にして「インストールしてすぐ・キー不要・無料・オフライン」で使えるようにする。上位の精度がほしい上級者だけが任意で自前キーを登録できる、2段構えにする。

- **段階1（既定）**: 端末内蔵のオンデバイス LLM。キー不要・無料・オフライン・完全プライベート。
- **段階2（任意・上級者向け）**: ユーザー自身の API キー（Claude / OpenAI / Gemini / Grok）でクラウドの上位モデルに切替。設定の「詳細」に置き、初回フローからは外す。

> 旧案にあった「サブスク（ChatGPT Plus 等）を VPS プロキシでブリッジする route」は廃止（非公開エンドポイント依存で規約・安定性のリスクが大きく、一般ユーザー向けの価値も薄いため）。

### 共通インターフェース（プロバイダ抽象化レイヤー）

各社・各実装（オンデバイス / クラウド）の差を吸収する 1 つの抽象クラスを切る。これで UI 側はプロバイダを意識せず呼べる。**オンデバイス実装もこの `RecipeProvider` に乗せる**（クラウド実装と同じインターフェース）。

```dart
abstract class RecipeProvider {
  String get displayName;   // "Claude", "Gemini" など
  String get model;         // モデル名
  Future<List<Recipe>> suggestRecipes(
      List<Ingredient> inventory, RecipeConstraints constraints);
  Future<Map<String, String>> normalize(List<String> names);
  // カメラ登録: 画像から食材候補を抽出（Vision 対応モデル）。確信度つきで返す
  Future<List<DetectedIngredient>> recognizeIngredients(List<ImageBytes> images);
}
// RecipeConstraints に「所有家電（appliances）」と「出力言語（outputLocale）」を含め、プロンプトに反映する。
// 例: ホットクック所有 → 自動調理メニュー（モード・時間）を提案 / 未所有 → 通常の鍋・フライパン手順
//     ヘルシオ所有 → ウォーターオーブンのメニュー・温度設定を含める
//     outputLocale=ja → 日本語で、en → 英語で title・steps を生成（JSON のキー名は固定）
// DetectedIngredient: { name, estimatedQuantity, confidence } を JSON で返させ、確認画面に渡す
```

> Vision について: オンデバイス・クラウドとも画像入力（マルチモーダル）対応の実装ならカメラ登録（食材認識）もキーなしで動く。Vision 非対応の状況（実装・モデル）ではカメラ登録を無効化する（`supportsVision` フラグを継続）。

### 段階1: オンデバイス AI（既定・キー不要・無料・オフライン）

OS 内蔵のオンデバイス LLM を `RecipeProvider` 実装として呼ぶ。platform channel で各 OS のネイティブ API を叩く。

| プラットフォーム | 使う仕組み | 実装クラス（予定） | 留意点 |
|---|---|---|---|
| iOS / iPadOS / macOS（26 以降） | Apple **Foundation Models framework**（オンデバイス LLM） | `AppleFoundationModelsProvider`（platform channel → Swift） | 構造化出力（`@Generable` / `@Guide`）で Recipe / DetectedIngredient を型安全に受け取れる。マルチモーダル対応でカメラ登録もキーなし。約3B の小型モデルのため上位クラウドモデルより精度は劣る。Apple Intelligence 有効が前提 |
| Android | **ML Kit GenAI Prompt API**（Gemini Nano / AICore） | `GeminiNanoProvider`（ML Kit GenAI） | 自然言語・マルチモーダルのカスタムプロンプトをオンデバイス処理（オフライン・無料）。**対応端末のみ**（Pixel 8 以降・Galaxy S24 以降など Gemini Nano 対応機）。コンテキスト長 約4,096 トークンと限られるため、在庫リストが長い場合は要約・分割して渡す。構造化出力は JSON を返させてパース |

- **Android 非対応端末の扱い**: Gemini Nano 非対応端末では AI 機能（献立提案・カメラ登録）を無効化し、その旨を案内する。在庫管理・買い物リスト連携など AI 非依存の機能は通常どおり使える。（将来の任意拡張として、非対応端末では段階2の自前キーでクラウドへ誘導する余地は残すが、MVP では無効化でよい）

### 段階2: 自分の API キー（任意・上級者向け）

「より賢い提案がほしい人だけ」が自分のキーを登録して上位のクラウドモデルへ切り替える。**既存のマルチプロバイダ設計をこの段階として残す**。設定画面の「詳細」内に置き、初回オンボーディングフローからは外す。

| プロバイダ | API route | 備考 |
|---|---|---|
| Claude | Anthropic Messages API（従量） | 精度重視のとき |
| OpenAI | OpenAI API（従量） | 献立提案は通常の gpt 系モデル |
| Gemini | Gemini API（**Flash 系は無料枠あり**） | クラウド側でコストを抑えたいとき |
| Grok | xAI API（OpenAI 互換・従量、**無料クレジットが手厚い**） | OpenAI 用クライアントをほぼ流用可能 |

### 既定プロバイダの解決ロジック

1. ユーザーが段階2で自前キーを登録し、そのプロバイダを選んでいれば、それを優先
2. それ以外は、オンデバイスが使える（iOS 26+ で Apple Intelligence 有効 / Android で Gemini Nano 対応）ならオンデバイスを既定にする
3. どちらも使えない（オンデバイス非対応かつ自前キーなし）→ AI 機能を無効化して案内

### 実装メモ

- **OpenAI と Grok は OpenAI 互換 API**（`OpenAiCompatibleProvider` で URL とキーの差し替えのみ）。**Claude / Gemini は REST を直接叩く**。レスポンス整形は共通の `Recipe` 型に寄せる。
- プロンプトと JSON パースは全実装（オンデバイス含む）で共有（`recipe_prompts.dart`）。レスポンスは **JSON 固定**（前置き・コードフェンス禁止）。オンデバイスは構造化出力 API があればそちらを優先しつつ、無い場合は JSON パースにフォールバック。
- 自前キーは **flutter_secure_storage**（macOS/iOS=Keychain / Android=Keystore）に保存。プロバイダごとに登録・切替できる設定（詳細）を用意。オンデバイス既定はキーを持たない。

## 課金・コスト戦略

- **既定はオンデバイス AI なので、ユーザーにも開発者にもランニングコストがゼロ**。キー取得も不要で、対応端末ならインストール直後から無料・オフラインで AI 機能が使える。全機能無料の方針（機能差なし）は維持。
- **段階2の自前キーは任意**。「より賢い提案がほしい上級者」がクラウドの上位モデルを使いたいときだけ、自分の API キーで従量課金（各自負担）。コスト最優先なら Gemini Flash 無料枠 / Grok の無料クレジットも選べる。
- アプリ本体は IAP もサブスクも持たない。ドネーション（Buy Me a Coffee 外部リンク）のみ。
- **Claude の年間契約が活きるのは開発そのもの**: この設計メモを Claude Code に渡してコードを書かせる工程でカバーされる。
- （廃止）VPS プロキシでサブスクをブリッジする route は検討から外した。

## Claude Code への渡し方

1. `tsukaikiri` フォルダのルートにこの設計メモを置く
2. `CLAUDE.md` を作り、一行「設計は ./設計メモ.md を参照」と書いて毎セッション読ませる
3. 一度に全実装させず、下の「開発の進め方」の順番で段階的に作らせる（各段階で動作確認）

### 渡す前チェックリスト（決定事項）

- **外部の手作業準備（Google Cloud / OAuth、Gemini API キー）は開発開始後に対応**。Claude Code の指示に従って都度取得する（先回りしない）
- **当面はシミュレータ中心で開発**。iOS 実機・リマインダー連携の実機確認が必要になった段階で Apple Developer 登録（年 $99）を行う
- **開発優先度: macOS → iOS → Android**。在庫 → 買い物リスト（リマインダー）→ AI まで macOS で縦に1本通し、次に iOS、その後 Android に着手
- **在庫の数量は「個数」を基本とし、単位をユーザー設定可能に**（調味料・米など量で表したいもの向け）。下記データモデル参照
- **アプリ ID は `com.futo4.tsukaikiri`** で確定（最初に設定）
- **カメラ登録は画像を縮小して送信・1回最大10枚**（Vision のコストガード）
- **オフライン時は実行 → 失敗時に「電波の良い場所か Wi-Fi へ」と表示**（事前判定での無効化はしない）
- **データ同期は iOS=iCloud / Android=Google Drive App Data**（ユーザー自身のクラウド・無料）
- **Buy Me a Coffee のアカウント / URL は後回し**。必要になってから作成し、リンクを差し込む

## 開発の進め方

1. `flutter create tsukaikiri` でプロジェクトを作成（macOS + iOS + Android ターゲット）
2. この設計メモをルートに置き、Claude Code に渡す（前回の Reminders CLI handoff と同じ流れ）
3. まず在庫管理の最小版を作る（ローカル DB のみ。買い物リスト連携も AI もなしで動く範囲）。**この段階で i18n の土台（flutter_localizations + ARB）も入れておく**（後付けより楽）
4. 買い物リスト連携を `ShoppingListService` 抽象で実装。まず macOS（EventKit リマインダー）から、次に iOS（共通コード流用）、最後に Android（Google Tasks）
5. プロバイダ抽象化レイヤーを作る。**既定はオンデバイス AI**（iOS/macOS=Apple Foundation Models / Android=Gemini Nano）。開発初期は実装の都合上クラウド1社（Gemini Flash 等）で土台を作ってもよいが、最終的な既定はオンデバイスに置く
6. 段階2の自前キー（OpenAI 互換の Grok/OpenAI はまとめて、Claude/Gemini は個別）を任意機能として追加。設定の「詳細」に置き、既定プロバイダ解決ロジック（自前キー優先→オンデバイス→無効化）で繋ぐ
7. 所有家電に応じた手順生成を後付け
8. カメラ登録（半自動）を追加。Vision で候補抽出 → 確認画面 → 在庫へ。献立提案の AI 実装ができてからの拡張なので最後でよい
9. データ同期（`SyncService` 抽象）を追加。iOS=iCloud / Android=Google Drive App Data。丸ごとバックアップ/復元から
10. ヘルプ / このアプリについて（データ出典表示）とドネーション（Buy Me a Coffee 外部リンク）をメニューに追加

外部連携（買い物リスト・AI）は後の段階に寄せることで、序盤は課金ゼロ・認証なしで開発・動作確認できる。
