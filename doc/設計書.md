## 1. アーキテクチャ概要

- **feature-first** のフォルダ構成。各 feature 内を `domain` / `data` / `presentation` の3層に分ける
- 状態管理は **Riverpod**。UI は provider 経由で repository を呼ぶ
- ローカル永続化は **Drift**（SQLite）。repository が Drift をカプセル化し、UI から DB 詳細を隠す
- OS 依存機能（買い物リスト・同期）は**抽象インターフェース**を1つ切り、OS ごとに実装を差し替える
- レイアウトは OS ではなく**画面幅**（`LayoutBuilder`/`MediaQuery`）で分岐

```
UI (presentation)  ──watch/read──▶  Riverpod Provider  ──▶  Repository  ──▶  Drift (AppDatabase)
                                          │
                              抽象IF (RecipeProvider 🚧 / ShoppingListService 🚧 / SyncService ✅ macOS)
```

## 4. データモデル

### 4.1 Drift スキーマ（✅ 実装済み）

**Ingredients**（生成データクラス名 `Ingredient`）
| 列 | 型 | 備考 |
|---|---|---|
| id | Text (PK) | UUID |
| name | Text | 表示名 |
| normalizedName | Text | 名寄せキー。AI連携前は name 流用 |
| category | Text(enum) | `textEnum<IngredientCategory>()`（列挙子名で保存）|
| quantity | Real | 小数可 |
| unit | Text | 定義済み=列挙子名 / カスタム=生文字列 |
| expiryDate | DateTime? | 任意 |
| updatedAt | DateTime | |

**SettingsTable**（生成データクラス名 `AppSettings`）— 単一レコード（id=0 固定）
| 列 | 型 | 備考 |
|---|---|---|
| id | Int (PK) | 既定 0 |
| localePref | Text | 'ja' / 'en' / 'system'（既定 'system'）|
| shoppingListId | Text? | 書き込み先リストの**識別子**（macOS/iOS=calendarIdentifier / Android=tasklist id）|
| shoppingListName | Text? | UI 表示用リスト名（識別子で引き当てた現在名）|
| selectedProvider | Text | 'gemini' / 'claude' / 'openai' / 'grok'（既定 'gemini'）|
| modelOverridesJson | Text | プロバイダごとのモデル上書き {providerId: modelId}（既定 '{}'）。選択肢は各社のモデル一覧 API から取得 |
| syncEnabled | Bool | 既定 false |
| lastSyncedAt | DateTime? | |
| appliancesJson | Text | 所有家電の JSON 配列（既定 '[]'）|

- `schemaVersion = 3`。スキーマ変更時は version をインクリメントし、`MigrationStrategy.onUpgrade` に `from < N` の分岐を追加する（v2: SettingsTable に連携系6列を追加 / v3: modelOverridesJson を追加）
- テスト時は `AppDatabase(NativeDatabase.memory())` でインメモリ注入

### 4.2 ドメインモデル（✅ 実装済み）

DB 行をそのまま UI に渡さず、settings 系はドメインモデルへ変換して扱う:

- **UserSettings**: localePref, shoppingListId/Name, selectedProvider, syncEnabled, lastSyncedAt, appliances[] — `SettingsRepository` が `AppSettings` 行から変換（appliancesJson は `List<Appliance>` にデコード）
- **Appliance**: type(hotcook/healsio), series?, capacity?, modelNumber? — JSON シリアライズ可能（appliancesJson に格納）
- **SuggestedRecipe / RecipeIngredient**: AI 提案の一時オブジェクト（title, ingredients[{name, amount}], appliance?, cookMode?, cookMinutes?, steps[], usesExpiringSoon）。**DB 保存は後フェーズ**
- **DetectedIngredient**: name, estimatedQuantity, unit, confidence(0.0〜1.0)
- **RecipeConstraints**: appliances[], outputLocale, extraRequest?, count(既定3)
- **ShoppingList / ShoppingListItem**: id+name / title+notes?

## 5. 抽象レイヤー（OS / プロバイダ差し替え）

### 5.1 RecipeProvider（AI 共通IF、2段構え: オンデバイス既定 + 任意の自前キー）

**方針（2026-06-13 転換）**: 一般ユーザーの API キー取得・入力は離脱要因のため、**オンデバイス AI を既定**（キー不要・無料・オフライン・プライベート）にし、自前キー（クラウド4社）は設定「詳細」の任意機能に降格する。両者とも同じ `RecipeProvider` 抽象に乗せ、UI は実装差を意識しない。詳細な背景は設計メモ「AI プロバイダ」を参照。


```dart
abstract class RecipeProvider {
  String get displayName;     // UI 表示名（例: 'Gemini'）
  String get modelId;         // 使用中のモデル ID
  bool get supportsVision;    // false なら recognizeIngredients は UnsupportedError
  Future<List<AiModel>> listModels();  // モデル一覧を API から取得（ハードコードしない）
  Future<List<SuggestedRecipe>> suggestRecipes(List<Ingredient> inventory, RecipeConstraints c);
  Future<Map<String, String>> normalize(List<String> names);  // {表示名 → normalizedName}
  Future<List<DetectedIngredient>> recognizeIngredients(List<Uint8List> images);
}
```
- 実装（優先順位順、全て ✅・REST 直叩き）: **Gemini**（`GeminiProvider`）→ **Grok**（`OpenAiCompatibleProvider.grok`、api.x.ai）→ **OpenAI**（`OpenAiCompatibleProvider.openai`）→ **Claude**（`ClaudeProvider`、Anthropic Messages API）。Grok と OpenAI は同一クライアントで URL+キー+モデルのみ差し替え
- **モデル名はハードコードしない**: `listModels()` で各社のモデル一覧 API（Gemini=`/v1beta/models`、OpenAI 互換=`/v1/models`、Claude=`/v1/models`）から取得し、ユーザーが設定で選択 → `UserSettings.modelOverrides` に永続化。実装に持つのは**一覧取得前・オフライン時のフォールバック既定値のみ**（gemini-2.0-flash / grok-4.3 / gpt-5-mini / claude-opus-4-8。2026-06 時点）
- **プロンプトとパースは全プロバイダで共有**（`recipe_prompts.dart`）。プロバイダを切り替えても挙動が変わらないようにする。コードフェンス・前置き混入に備え `extractJson` で防御的に抽出
- **API キーはコンストラクタ注入**。呼び出し元が `SecureStorageService` から取得して渡す（プロバイダはストレージに依存しない）
- **DI ファクトリ**: `createRecipeProvider(providerId, apiKey, model)`（`recipe_provider_factory.dart`）。Riverpod の `recipeProviderProvider` が selectedProvider・modelOverrides・API キーを解決して生成（キー未登録なら null）
- **タイムアウト 60 秒・自動リトライなし**（確定）。失敗時は UI から再試行（オフライン文言の方針と整合。重複した提案リクエストの防止）
- `RecipeConstraints` に `appliances` と `outputLocale` を持たせプロンプトへ反映
- Vision: 全4社の既定モデルが対応（`supportsVision=true`）。非対応モデル選択時の扱いは ❓
- JSON 出力の強制: プロンプト指示に加えて Gemini=`responseMimeType`、OpenAI/Grok=`response_format: json_object` で API 側でも強制（Claude はプロンプトのみ）。レスポンスは `utf8.decode(bodyBytes)` でデコード（charset 未指定サーバ対策）
- HTTP/レスポンス異常は共通の `RecipeProviderException(provider, statusCode, body)`（statusCode=0 は形式異常）
- 各メソッドの JSON スキーマ（確定）:
  - suggestRecipes: `{"recipes":[{"title","ingredients":[{"name","amount"}],"appliance","cookMode","cookMinutes","steps":[],"usesExpiringSoon"}]}`
  - normalize: `{"normalized":{"<元の名前>":"<キー>"}}`（キーは英小文字＋アンダースコア）
  - recognizeIngredients: `{"ingredients":[{"name","estimatedQuantity","unit","confidence"}]}`

#### 5.1.1 オンデバイス AI（段階1・既定、🚧 macOS/iOS 実装中）

OS 内蔵のオンデバイス LLM を `RecipeProvider` 実装として呼ぶ。クラウド実装との差を最小化するため、**ネイティブ側は「プロンプト文字列（+任意で画像）を受け取り、モデルに JSON を出力させてテキストを返す」薄いブリッジ**に徹し、プロンプト生成・JSON パース・スキーマは Dart 側の `recipe_prompts.dart`（`buildSuggestPrompt` / `buildNormalizePrompt` / `recognizePrompt` / `parse*`）を**完全再利用**する（クラウドと挙動・スキーマ完全一致）。

- **対象**: iOS / iPadOS / macOS（**26 以降**）。Apple **Foundation Models framework**（オンデバイス LLM、約3B、Apple Intelligence 有効が前提）。
- **platform channel** `com.futo4.tsukaikiri/ondevice_ai`（macOS=`MainFlutterWindow` で手動登録、iOS=同 Swift を流用）:
  - `availability()` → `{available: Bool, supportsVision: Bool, reason: String?}`。`SystemLanguageModel.default.availability` を判定（未対応 OS / Apple Intelligence 無効 / モデル未DL を reason で区別）
  - `generate(prompt: String, images: [FlutterStandardTypedData]?)` → モデル出力テキスト（JSON 期待）。`LanguageModelSession` に prompt を渡す
- **Dart 実装**: `OnDeviceAiService`（channel ラッパ）+ `AppleFoundationModelsProvider implements RecipeProvider`。`displayName='Apple Intelligence'` / `modelId='apple-foundation-models'` / `supportsVision` は availability の戻り値を注入。`listModels()` は単一のオンデバイスモデルを返す。失敗は共通 `RecipeProviderException('ondevice', …)`
- **Swift 側**: `OnDeviceAiPlugin.swift`。`#if canImport(FoundationModels)` でガードし、未対応 SDK/OS では `available=false` を返してコンパイル・実行とも安全に縮退
- **Vision**: Foundation Models のマルチモーダル可否を availability で報告し、`supportsVision=false` のときはカメラ登録を無効化（既存 `supportsVision` 機構を流用）
- **堅牢化（将来）**: 構造化出力（`@Generable` / `@Guide`）で型安全に受け取る案。MVP は「JSON テキスト → 既存 parser」で実装し、必要に応じて移行
- **Android（🚧 実装済み・実機生成は未検証）**: Dart は iOS/macOS と同じ `OnDeviceRecipeProvider`（channel・プロンプト・パース共通）。ネイティブは `android/app/.../OnDeviceAiPlugin.kt`（同 channel）が **AICore の `GenerativeModel.generateContent`**（Gemini Nano、`com.google.ai.edge.aicore:aicore`）を呼ぶ。`MainActivity.configureFlutterEngine` で登録。`availability` は AICore システムアプリ（`com.google.android.aicore`）の有無 + `SDK_INT>=34` を**軽量に**判定（モデル DL は伴わない）。生成は端末・DL 状況により失敗し得るため例外処理で `available=false` 相当に縮退。現状テキスト専用 = `supportsVision=false`。**AICore が minSdk 31 を要求するためアプリの minSdk を 31 に設定**。実機（Gemini Nano 対応端末）での生成品質確認はユーザー ToDo。
- **表示名**: `onDeviceDisplayName()`（`recipe_provider_factory.dart`）が `Platform.isAndroid`→'Gemini Nano' / それ以外→'Apple Intelligence' を返す。`OnDeviceRecipeProvider.displayName` と `providerDisplayInfo('ondevice')` が共用。
- **既定プロバイダ解決**（✅ 実装済み）: `recipeProviderProvider` が次の順で解決する — ①クラウド選択＆キー有り→そのクラウド ②上記以外（キー無し / `'ondevice'` 選択）でオンデバイス可→`AppleFoundationModelsProvider`（supportsVision は availability から）③どちらも不可→null（AI 無効）。`'ondevice'` は selectedProvider の sentinel 値として扱い、`supportedProviderIds`（factory が生成するクラウド4社）には含めない。オンデバイス可否は `onDeviceAiServiceProvider`（DI・差し替え可）の `availability()` で判定。`providerDisplayInfo('ondevice')` は 'Apple Intelligence' を返す。
  - **既定値（端末依存・初回起動時に決定）**: `SettingsRepository.initializeDefaultProviderIfUnset(id)` が設定行未作成のときだけ `selectedProvider` を保存する。`main()` で `UncontrolledProviderScope` 化し、起動時に `onDeviceAiServiceProvider.availability()` を見て **対応端末→`'ondevice'` / 非対応→`'gemini'`** を渡す。以降はユーザー選択を尊重（初回のみ）。
  - **プロバイダ選択 UI**（desktop `_ProviderGrid` / mobile `ai_settings_screen`）: 先頭にオンデバイス（Apple Intelligence）を表示。`onDeviceAiAvailabilityProvider`（FutureProvider）で可否を取得し、非対応端末はグレーアウトして選択不可。`selectedProvider=='ondevice'` のときは API キー欄・モデル欄を隠す（キー不要）。
  - **オンボーディング AI ステップ**（desktop `_AiStep` / mobile `_MobileAiStep`）: 初回フローでは **API キー入力・プロバイダ選択をしない**。`onDeviceAiAvailabilityProvider` を見て可否のみ表示（可=「{name} が端末内で動作します…」/ 不可=「設定からご自分の API キーを登録すればクラウド AI を使える」案内、共有 `OnDeviceStatusCard`）。完了サマリーの AI 名は `_resolveAiName` が `'ondevice'`→`onDeviceDisplayName()` を返す。旧 `OnboardingProviderGrid`/`OnboardingApiKeyCard` は未使用化（将来削除）。

### 5.2 ShoppingListService（買い物リスト共通IF、🚧 macOS 実装済み・UI 未接続）
```dart
getLists() / createList(name) / addItems(listId, items)  // addItems は実追加件数を返す
```
- macOS/iOS=**自前 platform channel で確定**（既存プラグインは不採用）。channel 名 `com.futo4.tsukaikiri/reminders`、Swift 側は `RemindersPlugin.swift`（EventKit は macOS/iOS 共通のため iOS にも同実装を流用可）。`MainFlutterWindow.swift` で手動登録
- Dart 側実装 `RemindersShoppingListService` は IF 外に `requestAccess()` を持つ（初回オンボーディングで権限リクエスト）
- **重複チェック（確定）**: 追加先リストの**未完了**リマインダーとタイトルを**小文字化して完全一致**比較し、重複はスキップ。戻り値は実際に追加した件数
- `createList` は iCloud ソース優先、なければローカルソースに作成
- Android=Google Tasks REST（`auth/tasks`）🔜
- 保存は**識別子**（calendarIdentifier / tasklist id）

### 5.3 SyncService（同期共通IF、✅ macOS iCloud 実装済み 2026-06-11）
- IF: `isAvailable()` / `writeBackup(payload)` / `readBackup()`（`lib/features/sync/service/sync_service.dart`）。例外は `SyncNotSignedInException` / `SyncIOException`
- macOS/iOS = **iCloud ubiquity コンテナ**で確定。`ICloudSyncService`（`icloud_sync_service.dart`）が platform channel `com.futo4.tsukaikiri/icloud_sync` ↔ `ICloudSyncPlugin.swift` を呼ぶ。保存先はコンテナ `iCloud.com.futo4.tsukaikiri` の隠し領域 `backup/tsukaikiri_backup.json`（単一ファイル上書き = last-write-wins。`Documents/` には置かずユーザーから不可視）
- Swift 側: コンテナ解決はバックグラウンドスレッド、読み書きは `NSFileCoordinator`。未ダウンロード時はプレースホルダ（`.〜.icloud`）を検出して `startDownloadingUbiquitousItem` → 最大10秒ポーリング（新しい Mac での初回復元）
- **バックアップ形式**: スキーマ版付き JSON（`formatVersion` / `appDbSchemaVersion` / `exportedAt` / `ingredients[]` / `settings`）。`BackupCodec`（`backup_codec.dart`）が encode/decode と検証（未来バージョンは専用エラー）。**API キーは含めない**（iCloud キーチェーン同期 = `flutter_secure_storage` の `synchronizable: true` が担当。旧アイテムからの自動移行+開発署名フォールバックあり）
- **トリガ**: 自動（`autoBackupWatcherProvider` が在庫・設定の stream を購読 → `BackupScheduler` で5秒デバウンス → `silentBackup()`。`syncEnabled` トグルで ON/OFF）+ 手動ボタン。復元は手動+確認ダイアログのみで、復元中は `_isRestoring` で自動バックアップを抑止
- **ループ防止（重要）**: バックアップが `lastSyncedAt` を更新すると設定 stream が発火するため、設定側は lastSyncedAt を除いた指紋の変化時のみスケジュールする（指紋比較がないとバックアップが永久ループする）
- `SyncController` の状態は種別 enum（`SyncSuccessKind` / `SyncErrorKind`）で持ち、表示文言は view が l10n で解決（i18n 規約）
- Android = Google Drive App Data（`drive.appdata`）🔜 — 同じ JSON 形式を使い回す

## 7. 主要ロジック

### 7.1 在庫ソート（InventoryRepository）
SQLite の `ORDER BY expiryDate IS NULL, expiryDate ASC, updatedAt DESC`
→ 期限なしを末尾、期限近い順、同順は更新新しい順。

### 7.2 数量の増減（increment / decrement）
`max(0, quantity ± by)` に更新し updatedAt 更新。**0 でも行は削除しない**。詳細画面のステッパーと一覧操作で使用。
「使い切った」「削除」は明示削除（`deleteById`）で、UI はスナックバーの「元に戻す」で取り消し可能。

### 7.3 期限警告（expiry_status.dart）
カレンダー日で残り日数を算出 → `<0:expired / 0:today / 1〜3:warning / 4+:normal / null:none`。色は ColorScheme に対応付け。

### 7.4 日持ち初期値（shelf_life.dart / shelf_life_table.dart）

**データパイプライン（ビルド時生成・実行時 API 非依存）**:
- 入力① `tool/shelf_life/foodkeeper_ingredients.csv`: USDA FoodKeeper 2019-06 のミラー（`jelera/food-shelflife-db`・663 行・CC0）を固定値としてコミット。公式配布元（fsis.usda.gov）は開発環境から 403 で取得不可のためミラーを採用。差し替え手順は `tool/shelf_life/README.md`。
- 入力② `tool/shelf_life/ja_mapping.json`: 人手レビュー対象。`foodkeeper`（FoodKeeper 行を `Name`+`Name_subtitle` 部分一致で特定し日本語エイリアス・normalizedKey・category を付与・97 項目）+ `supplement`（FoodKeeper に無い和食材を日数直指定・94 項目）。
- 生成 `tool/shelf_life/generate.dart`（純 Dart CLI・コアは `shelf_life_compute.dart` に分離）: 冷蔵中央値 `round((min+max)/2)`・Metric 換算（Days=1/Weeks=7/Months=30/Years=365）・保存源フォールバック（Refrigerate→DOP_Refrigerate→Pantry→DOP_Pantry）・数値化不能 Metric はスキップ。日本語エイリアスを持つ項目のみ出力。**冪等**。出力 `assets/shelf_life/shelf_life_rules.json`（schemaVersion/normalizedKey/aliases/category/days/source の安定スキーマ・185 ルール）。

**実行時**:
- `ShelfLifeTable`（`shelf_life_table.dart`）: 同期 `daysFor(name, {category})`。照合 ①エイリアス完全一致→②部分一致（最長優先）→③normalizedKey→④null。`load()`（rootBundle）/ `empty()`。`shelfLifeTableProvider`（既定 empty）を `main()` の起動時 load → ProviderScope overrides で注入（失敗時 empty で起動継続）。
- `expiryFromName(table, name, category, from)`: テーブルヒット→その日数、ミス→`defaultExpiryFrom`（カテゴリ別フォールバック）。期限プレフィル（食材フォーム・カメラ確認）はこの関数経由。
- `defaultExpiryFrom(category, from)`: カテゴリ目安日数を加算。目安なし（other）は null。テーブルにヒットしないときの最終フォールバックとして残す。

### 7.5 単位表示（unit_option.dart）
`unitLabel(token, l10n)`: token が `UnitOption` の列挙子名ならローカライズ、なければカスタムとして生表示。保存値は言語非依存。

### 7.6 プロンプト設計（recipe_prompts.dart · buildSuggestPrompt）
- 在庫は「名前 数量単位 ＋ 期限注記」の箇条書きで渡す。残り2日以内は **【期限間近N日】** と強調し、優先消費を指示
- 所有家電セクション: 家電なしは「（なし）」。家電ありは **種別+容量** の箇条書き（`- hotcook（容量 2.4L）` / capacity が null なら `- hotcook`）。シリーズ・型番は渡さない
- 家電あり時は条件セクションに以下の指示を追加（ハルシネーション対策・フェーズ6確定方針）:
  - 所有家電での調理が向く料理には家電手順を steps に含める（**汎用的な調理モード名と加熱時間**を明記）
  - 向かない料理は通常の鍋・フライパン手順でよい（全レシピを家電に寄せない）
  - 機種固有の自動メニュー名・メニュー番号は出力しない
- cookMode/cookMinutes フィールドは JSON スキーマに温存（パース・表示は変更なし）。調理モード・時間は steps の文中にも自然に含める
- 「返答は JSON 形式のみ。前置き・コードフェンス・説明文は禁止」を明記し、`responseMimeType`（Gemini）/ `response_format`（OpenAI互換）でも JSON を強制（二重ガード）
- フィールド名は言語固定、自然文（title・steps 等）のみ `outputLocale` の言語で生成させる
- `extraRequest`（「あと1品」等）と `count`（提案件数）をプロンプトに反映

### 7.7 設定の upsert（settings_repository.dart）
フィールド単位の setter（`setLocalePref` / `setShoppingList` / `setSelectedProvider` 等）→ 内部 `_upsert` が既存行を読み出し、指定フィールドだけ上書きして `insertOnConflictUpdate`。行が無ければ既定値で新規作成。

### 7.8 不足材料の抽出（missing_ingredients_service.dart、確定）
- 純関数 `findMissingIngredients(recipes, inventory, normalizedNames)`: 在庫扱いは**数量 > 0** のみ。判定は ①名前一致（trim+小文字化）→ ②名寄せキー一致 の順
- 名寄せキーが同じ材料は同一視し、由来献立ごとの分量を `MissingIngredient.sources` に集約（表示名は最初に出現した献立での表記）
- `MissingIngredientsService.find()` が献立側の材料名を `RecipeProvider.normalize` で名寄せしてから突き合わせる（表記ゆれ吸収）
- **買い物リストへの数量の載せ方（確定）**: `toShoppingListItem()` で **タイトル=材料名のみ**、**notes=「献立名: 分量」の連結**。リマインダー側の重複チェックがタイトル完全一致のため、タイトルに分量を含めない

### 7.9 normalizedName バックフィル（normalized_name_backfill_service.dart、確定）
- 対象は `normalizedName == name` の行（AI 連携前に name を流用した行）。`InventoryRepository.findUnnormalized` / `applyNormalizedNames` を使用
- 名前を重複排除し 50 件ずつ `normalize` に渡す。キーが名前と同一なら書き込まない。**updatedAt は変更しない**（ユーザー編集ではないため。並び順・将来の同期判定に影響させない）
- **冪等**（付与済みには触れず、対象ゼロなら AI を呼ばない）。実行タイミングは献立提案フロー実行後のバックグラウンドを推奨（UI 接続時に配線）

## 9. 規約・方針

- **主キーは UUID(String)**（同期競合回避）
- **列挙子名・normalizedName は言語非依存キー**として保存（表示言語が変わっても不変）
- AI 出力は JSON 固定でパース、確認画面必須（カメラ）
- **API キーは DB に入れない**。`SecureStorageService`（キー形式 `api_key_<provider>`）に保管し、プロバイダにはコンストラクタで注入
- **DB スキーマ変更時**は `schemaVersion` をインクリメントし、`onUpgrade` に `from < N` 分岐を追加
- 日持ちデータは実行時に API を呼ばずビルド時同梱
- レイアウトは画面幅で分岐（`_tabletBreakpoint = 720`：狭い=単一ペイン / 広い=二ペイン master-detail）
